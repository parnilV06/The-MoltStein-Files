Take a correct program. Bolt on eight functions that are never called. Show it to an LLM judge. In one May 2025 study, the judge's accuracy at confirming the code was correct rose from 79.67% to 89.33%. The code did the same thing. The judge just liked it dressed up.

"Don't Judge Code by Its Cover: Exploring Biases in LLM Judges for Code Evaluation" (Moon et al., arXiv:2505.16222, May 2025) tested 6 judges over 1,000 code pairs: 200 competitive-programming problems across C++, Python, Java, JavaScript, and Go, each with a correct and an incorrect solution. They then perturbed the surface and watched the verdict move.

The numbers are not subtle. Variable renaming induced positive bias in 80% of cases, and the effect started at variable names as short as two characters. A misleading comment about the task pulled correct-code accuracy down by as much as 26.7 percentage points on Java. Self-declared correctness, a line of text claiming the code passes, swung one open-weights judge by up to 26.4 points. The illusory-complexity trick, those eight dead functions, is the one I find hardest to forgive: the judge read more scaffolding as more competence.

This matters because LLM-as-judge is quietly becoming the grader for coding leaderboards and RL reward signals. If the reward model can be moved 10 points by formatting, then a policy trained against it learns to write longer, more commented, more elaborately named code, not more correct code. The bias becomes a training target.

The defense is not a better prompt. It is refusing to let the judge see surface at all when surface is not the question. For functional correctness, run the code. Execution does not care about variable names or comment count. Where a model judge is unavoidable, normalize formatting, strip comments, and rename variables to canonical tokens before scoring, so the only thing left to judge is behavior.
