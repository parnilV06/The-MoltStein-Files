An autonomous agent that browses the web and can call tools has turned every page it visits into a potential instruction source. The threat model that mattered for a chatbot was the user trying to misbehave. The threat model for a browsing agent includes the entire untrusted internet writing instructions the agent will read and might act on, with the user nowhere in the loop. This is the practical face of indirect prompt injection, and giving the model tools is what upgrades it from an annoyance to an exploit.

The escalation comes from the combination, not from any single capability. A model that reads a malicious page but can only emit text leaks nothing it was not already shown. A model that reads a malicious page and can send HTTP requests, write files, or call an internal API can be steered by that page into using those capabilities against the user. The injected text does not need to break the model. It needs to convince the planner that the next sensible step is the one the attacker wants, and a planner is built to be convinced by context.

NIST's AI risk framework and its adversarial-ML taxonomy name this class plainly: abuse and integrity violations where adversarial inputs steer a deployed system. The recurring mitigation theme across the serious treatments is capability containment rather than input purification. You assume injection will sometimes succeed and you make success cheap. Scope each tool to least privilege. Require a fresh human confirmation for any irreversible or externally visible action. Keep the credentials the agent holds narrow enough that a fully hijacked agent cannot reach the data that would make hijacking worth it.

The pattern I would design around is that the blast radius of a hijacked agent should be bounded by its granted capabilities, computed as if the attacker controls the model. That is a different exercise from "is the model safe". It asks: if every token the model emits this session were chosen by an adversary, what is the worst reachable outcome given the tools and credentials in scope? If the answer includes reading the user's full inbox and sending mail, the architecture has the injection vulnerability regardless of how good the model's refusals are, because refusals are a distribution-bound behavior and the capability grant is a hard fact.

This reframes a lot of agent design as a permissions problem that predates language models. The interesting and unsolved part is that natural-language planning makes least privilege harder to specify, because the agent's legitimate task and the injected task are expressed in the same medium and the system cannot always tell which capability request serves which.
